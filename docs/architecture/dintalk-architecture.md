# 钉钉机器人协作集成架构

> 将现有 Web 端 AI 协作流程扩展到钉钉群，使纯 IM 用户也能完成全部协作操作。
> 设计原则：**协作内核零改动，钉钉只是一层适配器。**

## 一、背景与目标

### 现状
当前协作流程基于 Web + Socket.IO，数据写入 `transcripts/{projectId}/{draftId}/discussion.json`（见 `packages/server/src/ai/transcript.ts`）。用户必须打开浏览器才能参与讨论、召唤 AI（`@助手` 前缀）、查看卡片。

### 目标
让用户在钉钉群内通过机器人完成以下全部协作操作：

1. 发起讨论（创建 draft）
2. 发言 / 追问
3. 召唤 AI
4. 查看 AI 结构化产出（卡片）
5. codingLock 锁定 / 释放
6. 查看历史
7. Web 与钉钉实时双端同步

### 非目标
- 不替代 Web 端。富交互（时间线视图、diff 渲染、画布等）仍以 Web 为主。
- 不做跨组织 / 跨企业的机器人。

## 二、核心映射模型

```
DingTalk 群  1 : 1  Draft
     ↑            ↑
     └── 所属 ──→ Project (隐式)
```

**关键语义**：群即语境。群内所有消息默认属于该 draft，`projectId` 和 `draftId` 由 `groupId` 反查绑定表得到，用户不需要显式指明。

**为什么不用"一个群一个项目"？**
- 简化用户心智：发言前无需选择当前是在讨论哪个 draft。
- 切换 draft = 切换群，符合钉钉肌肉记忆。
- 成员粒度按 draft 控制，符合"讨论参与者 ≠ 项目所有成员"的现实。

**"群 = Draft" 的成本**：
- 新建 draft 需要建群（较重）→ 通过 `/新建` 命令自动化解决。
- 跨 draft 可见性降低 → 仍走 Web 查看全貌。

> **备选方案**：若钉钉开放平台的话题（Topic/Thread）回调已携带 `topicId`，可升级为"群 = Project + 话题 = Draft"，新建更轻、同群纵览更强。实施前需验证该 API。

## 四、数据模型

### 绑定表 `dingtalk_bindings`

| 字段 | 类型 | 说明 |
|------|------|------|
| groupId | string | 钉钉群 ID（主键） |
| projectId | int | 项目 ID |
| draftId | int \| null | Draft ID；`role='project_home'` 时为 null |
| role | enum | `project_home` \| `draft` |
| parent | string \| null | 父群 groupId（仅 draft 群使用，追溯来源） |
| createdBy | string | 发起绑定的钉钉 unionId |
| createdAt | timestamp | |

### 身份绑定表 `dingtalk_users`

| 字段 | 类型 | 说明 |
|------|------|------|
| unionId | string | 钉钉 unionId（主键） |
| userId | int | 系统内用户 ID |
| boundAt | timestamp | |

### 项目成员表 `project_members`

| 字段 | 类型 | 说明 |
|------|------|------|
| projectId | int | 复合主键 |
| userId | int | 复合主键 |
| role | enum | `owner` / `editor` / `viewer` |
| joinedAt | timestamp | |

**鉴权独立于钉钉群**——群成员是否能操作项目，完全看这张表，不看他在哪个群里。

未绑定身份的用户发言入库但标记 `unverified`，不触发 AI、不可点击 ActionCard 按钮。

## 七、鉴权设计

鉴权拆成两个独立层，**不能混为一谈**：

| 层 | 解决的问题 | 载体 |
|----|-----------|------|
| **认证 (Authentication)** | unionId 对应哪个系统 userId | `dingtalk_users` 表 |
| **授权 (Authorization)** | 这个 userId 在项目内能做什么 | `project_members` 表 |

**核心原则**：**钉钉群成员 ≠ 项目成员**。授权完全由 `project_members` 决定，不看他在哪个群里。否则一个被误拉进群的人就能看到/修改他不该看的东西。

### 7.1 认证：钉钉免登 OAuth

用钉钉企业内部应用自带的"免登"能力——H5 在钉钉容器内打开时可静默拿到 `authCode`，后端换 `unionId`。这是业内标准做法，用户体验最好，不走其它备选方案。

```
  用户        钉钉群         机器人         adapter      钉钉OAuth     Web端登录   DB
    │           │              │              │             │            │        │
    │ 首次发言/点按钮                                                              │
    ├──────────▶│              │              │             │            │        │
    │           ├─────────────▶│ 查 unionId   │             │            │        │
    │           │              ├─────────────▶│             │            │        │
    │           │              │              │ 查 dingtalk_users         │        │
    │           │              │              ├─────────────────────────────────▶│
    │           │              │              │◀─────── 未绑定 ──────────────────┤
    │           │◀──ActionCard "点此绑定账号" │             │            │        │
    │ 点击      │              │              │             │            │        │
    ├──────────▶│              │              │             │            │        │
    │           │ 跳 H5 (钉钉容器内打开)      │             │            │        │
    │◀─────────────────────────────────────────▶│          │            │        │
    │ H5 自动获取 authCode                     │             │            │        │
    │           │              │              │ 换 unionId  │            │        │
    │           │              │              ├────────────▶│            │        │
    │           │              │              │◀─ unionId ──┤            │        │
    │           │              │              │                                   │
    │ H5 提示"请登录系统账号" → 输入账密/SSO   │                                   │
    ├─────────────────────────────────────────────────────▶│                    │
    │◀─ userId ──────────────────────────────────────────  │                    │
    │           │              │              │ 写绑定 (unionId → userId)         │
    │           │              │              ├─────────────────────────────────▶│
    │           │              │              │                                   │
    │◀─ "✓ 绑定成功" ─────────────────────────┤                                   │
```

**体验点**：
- 一次绑定全局生效，所有群都自动识别。
- 绑定失败时机器人 DM 给出明确诊断（否则用户以为机器人坏了）。

**未绑定用户策略**：消息**入库但标记 `unverified`**，不触发 AI、不可点击 ActionCard 按钮。机器人在首条 unverified 消息后主动弹绑定卡片。

### 7.2 授权：独立 ACL

每条入站消息的鉴权判定顺序：

```
  1. unionId 是否已绑定 userId？                否 → unverified 策略
  2. userId 是否是 binding.projectId 的成员？   否 → 拒绝 + 提示申请加入
  3. userId 的 role 是否允许本次操作？           否 → 拒绝 + 提示角色
  4. 放行 → 调用 execution-manager / transcript
```

**操作 × 角色矩阵**：

| 操作 | viewer | editor | owner |
|------|:------:|:------:|:-----:|
| 查看讨论历史 | ✓ | ✓ | ✓ |
| 普通发言 | ✓ | ✓ | ✓ |
| 召唤 AI（`@机器人`） | ✗ | ✓ | ✓ |
| 建 draft（`/新建`） | ✗ | ✓ | ✓ |
| codingLock 锁定 | ✗ | ✓ | ✓ |
| codingLock 强制释放他人 | ✗ | ✗ | ✓ |
| 归档 draft | ✗ | ✗ | ✓ |
| 把新人加入项目 | ✗ | ✗ | ✓ |

### 7.3 群成员非项目成员的处理

问题场景：有人被拉进钉钉群，但 `project_members` 里没有他。机器人对他的 @ 回复：

> 你不是项目 XX 的成员，只能围观。申请加入？[申请]

他的普通发言仍会入库（群本身是开放的），但任何需要写操作或触发 AI 的动作都被拒绝。

### 7.4 安全硬化项

1. **钉钉回调签名校验**（必做）——HMAC-SHA256 校验 `timestamp + sign`，timestamp 偏差超过 5 分钟拒绝（防重放）。否则任何人都能伪造 POST。
2. **ActionCard 回调身份不信任 payload**——卡片按钮回调时，永远以 webhook 实体 `unionId` 为准，不要读按钮 payload 里编码的用户身份。
3. **codingLock 抢占**——校验 `role + 当前状态`；editor 只能锁"空闲"或自己的锁；owner 可强制释放他人。
4. **绑定表保护** `dingtalk_users` / `dingtalk_bindings`：
   - 全量审计日志（谁在何时绑定/解绑了哪个 unionId）
   - 每个 userId 最多绑 1 个 unionId，反之亦然
   - 解绑需二次确认

### 7.5 明确不做的事

- **AI 调用的配额与防刷**——用户使用自己的 AK 执行，成本由用户自担，不需要系统限流。
- **项目成员变更时的实时同步**——owner 在 Web 端移除某成员后，其权限在下一条消息到达时失效即可，不做即时推送。

### 7.6 分阶段落地

**Phase A（MVP）**
- 钉钉免登 OAuth 绑定
- `project_members` 三级角色 ACL
- `unverified` 降级策略
- Webhook 签名校验 + 重放保护

**Phase B（治理完善）**
- 非项目成员"围观 + 申请加入"机制
- 绑定操作审计日志
- codingLock 抢占策略（editor 软规则 + owner 强制）

## 八、风险与悬而未决的问题

### 钉钉平台层面

1. **消息回调触发条件**：群机器人只在被 @ 时收到回调。这意味着"未@机器人的普通发言"默认不入库。
   - 可选解：要求所有协作发言都带 @；或申请企业消息监听权限（需企业认证 + 审核）。
2. **建群 API 权限**：程序化建群要求企业内部应用，且 corp 级授权。SaaS 部署下每家企业需单独配置。
3. **互动卡片 2.0 可用性**：流式卡片需要验证在目标钉钉版本的兼容性。
4. **话题（Topic）回调字段**：若想升级到"群=Project + 话题=Draft"模型，必须先验证回调中是否包含 `topicId`。

### 身份与权限

1. **身份双轨**：已在第七节定义——钉钉免登 OAuth 绑定 + `project_members` 独立 ACL。未绑定用户降级为 `unverified` 只读旁观者。
2. **退群 / 解散**：draft archived 后如何处理钉钉群？建议改名 `[已归档]` + 收回机器人权限，避免直接解散。

### 富交互降级

1. Web 端 timeline / split / canvas 三种视图在钉钉内退化为线性消息流。
2. 代码 diff 依赖 markdown 渲染，复杂场景仍需跳转 Web。
3. 长历史检索：群聊自带搜索，但跨群 / 跨时间段检索仍要走 Web。

### 并发与一致性

1. `transcript.ts` 文件写入的并发风险在双端接入后会放大（Phase 2 已标注需要文件锁）。
2. Web 与钉钉同时编辑同一卡片时的冲突策略待定。

## 九、最小落地路径

1. **新建模块** `packages/server/src/modules/dingtalk/`：
   - `bindings.repository.ts` — 绑定表 CRUD
   - `adapter.ts` — 入站路由 + 出站卡片渲染
   - `webhook.controller.ts` — 钉钉回调入口
   - `openapi.client.ts` — 建群 / 发卡片客户端

3. **出站**：订阅 Socket.IO 频道 → 按绑定表反查 groupId → 推互动卡片。
4. **首个里程碑**：跑通"流程 3（日常对话 + AI 响应）"，其它流程后续迭代。
